# The Cost Model — Why 7x and How to Design Around It

Agent teams are not expensive by accident. They are expensive by design — each teammate runs a full context window, accumulates its own history, and makes its own API calls independently. Understanding what drives the cost is the first step to controlling it.

---

## The Four Cost Drivers

### Driver 1 — Model Selection (Largest Lever)

Already covered in [Section 1](../01-model-selection/cost-implications.md). The short version:

- 7x token multiplier × Opus rate ≈ **21x baseline cost**
- 7x token multiplier × Sonnet rate ≈ **7x baseline cost**
- 7x token multiplier × Haiku rate ≈ **2–3x baseline cost**

Sonnet for teammates. Haiku for high-volume, well-scoped workers. Opus only for orchestrators on genuinely complex pipelines. This decision dwarfs every other cost lever.

---

### Driver 2 — Context Window Accumulation Per Agent

This is the one that surprises practitioners. Token costs don't grow linearly with turn count — they compound, because every turn includes all prior turns as input.

**The compounding math:**

```
Turn 1:   500 tokens input
Turn 2:   500 + 500     = 1,000 tokens input
Turn 3:   500 + 500 + 500 = 1,500 tokens input
Turn N:   N × 500 tokens input

Total input tokens = 500 × (1 + 2 + 3 + ... + N) = 500 × N(N+1)/2
```

| Turns | Naive estimate | Actual input tokens | Actual/naive |
|-------|---------------|---------------------|-------------|
| 5     | 2,500         | 7,500               | 3x          |
| 10    | 5,000         | 27,500              | 5.5x        |
| 20    | 10,000        | 105,000             | **10.5x**   |
| 30    | 15,000        | 232,500             | **15.5x**   |

At 20 turns, a developer who estimated cost from "tokens per turn × turns" will be off by 10.5x before model rates enter the calculation. At 30 turns, 15.5x.

On a 5-agent team each running 20 turns: 5 × 105,000 = **525,000 input tokens** — from 500 tokens of actual content per turn. This is why the 7x figure is a floor, not a ceiling.

**The implication for agent design:** Long-running agents are disproportionately expensive. An agent that runs 30 turns costs 15x more than a naive estimate. Design tasks to terminate early. Decompose long tasks into shorter subtasks with fresh context.

---

### Driver 3 — Tool Result Size

Covered in [Section 1](../01-model-selection/cost-implications.md) and [Section 2](../02-errors-and-stop-reasons/recovery-patterns.md). Compounds with Driver 2 — large tool results added to a long-running context multiply the accumulation effect.

Truncate and summarize tool results before returning. This is not optional on long-running agent teams.

---

### Driver 4 — Team Size

Each additional teammate is a full context window at full model rate. The per-agent compounding from Driver 2 scales with team size. A 10-agent team accumulating 20 turns each is 10 × 105,000 = 1,050,000 input tokens from 500 tokens of actual content per turn.

Do not add teammates speculatively. Every teammate needs a defined role that justifies its cost. Adding a "just in case" agent to a team is adding a full context window.

---

## Context Compaction — The Architectural Answer

Auto-compaction summarizes conversation history when context approaches the limit, allowing long-running agents to continue without hitting the 413 or runaway accumulation costs. Each agent in a team manages compaction independently.

**When to trigger compaction manually** (before auto-compaction forces it):

```python
def should_compact(client, model, messages, tools=None, threshold=0.75) -> bool:
    """
    Check if context is approaching compaction threshold.
    Trigger at 75% to compact on your schedule, not the API's.
    Compacting at 75% preserves more fidelity than emergency compaction at 95%.
    """
    response = client.messages.count_tokens(
        model=model,
        messages=messages,
        tools=tools or []
    )
    context_limits = {
        "claude-opus-4-6":          200_000,
        "claude-sonnet-4-6":        200_000,
        "claude-haiku-4-5-20251001": 200_000,
    }
    limit = context_limits.get(model, 200_000)
    return (response.input_tokens / limit) >= threshold


def compact_messages(messages: list, keep_recent: int = 6) -> list:
    """
    Minimal compaction: keep system message and N most recent turns.
    For production: replace the pruned middle with a summary turn
    generated by calling the model on the history being removed.
    """
    system = [m for m in messages if m.get("role") == "system"]
    non_system = [m for m in messages if m.get("role") != "system"]
    recent = non_system[-keep_recent:] if len(non_system) > keep_recent else non_system
    return system + recent
```

**Compaction tradeoff:** Compacting earlier preserves more context fidelity. Emergency compaction at 95% discards more, faster, and may drop important prior context. The 75% threshold is intentional — compact before the crunch, not during it.

## Real-World Cost Reference

| Configuration | Estimated monthly cost (developer-level usage) |
|--------------|------------------------------------------------|
| Agent teams, Sonnet 4.6 | ~$100–200/month (Anthropic documented) |
| Agent teams, Opus 4.6 | ~$300–600/month (estimated 3x multiplier) |
| Agent teams, Haiku 4.5 workers | ~$30–60/month (estimated 60–70% reduction) |
| 8-agent team, 13 disciplined runs, prompt caching enabled | $16.46 (observed — achievable with mixed model assignment) |

The $16.46 figure is not theoretical. It comes from a real multi-agent run with disciplined model assignment (mixed Sonnet/Haiku workers, not Opus across the board) and prompt caching active. It demonstrates what the cost model looks like when all the levers are used correctly.
